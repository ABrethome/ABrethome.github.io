**Title: Newsletter March 2026 - L'IA est-elle compatible avec nos objectifs climatiques ? [Is AI compatible with our climate goals?]**

[Link (to AI report, not the newsletter)](https://theshiftproject.org/app/uploads/2025/04/2025_03_06-TSP-Rapport-intermediaire-IA-quelles-infra-num-monde-decarbone.pdf)

Summary: Generative AI’s climate impact often appears negligible at the individual level (e.g. ~1.14 gCO₂e for a 400‑token query in MistralAI) but is structurally significant at scale. More than 50% of data‑center electricity is still fossil‑based, making emissions 10-20x higher than with low‑carbon power. Hardware manufacturing contributes ~25% of total emissions, and frequent equipment renewal adds pressure. Despite efficiency gains, exploding demand has accelerated data‑center energy use from ~7% annual growth (2014–2019) to ~13% (2019–2024). By 2030, emissions could reach ~920 MtCO₂e/year—around twice France's annual emissions. Climate compatibility requires simultaneous action: decarbonizing electricity, reducing manufacturing footprints, and capping total electricity use, with an estimated upper limit around ~1000 TWh.

**Title: Security Best Practices**

[Link](https://modelcontextprotocol.io/docs/tutorials/security/security_best_practices)

Summary: Best practices for MCP servers:
- Implement per‑client user consent before forwarding any request to third‑party authorization servers to avoid “confused deputy” attacks.
Never rely on third‑party consent cookies alone; always enforce MCP‑level consent tied to each client_id.
Maintain a secure registry of approved client IDs per user and check it on every authorization request.
Use a clear MCP‑owned consent UI that identifies the client, requested scopes, and registered redirect URI.
Protect consent flows against CSRF and clickjacking using state tokens, CSP frame-ancestors, or X-Frame-Options.
Strictly validate redirect URIs using exact string matching; reject unregistered or modified URIs.
If cookies are used for consent state, secure them with __Host- prefix, Secure, HttpOnly, SameSite=Lax, and bind them to the specific client.
Store consent decisions securely server‑side (database or signed sessions), not in client‑controlled state.

**Title: Top 20 AI Projects on GitHub to Watch in 2026: Not Just OpenClaw**

[Link](https://medium.com/@nocobase/top-20-ai-projects-on-github-to-watch-in-2026-not-just-openclaw-909b3bae62f6)

Summary: Key take aways for me from this list:
- **OpenClaw:** OpenClaw represents the shift from “chatbot apps” to always‑on personal AI gateways. Instead of forcing users into a single UI, it embeds AI into existing messaging channels and devices, emphasizing self‑hosting, plugin extensibility, and persistent agent execution. It signals a move toward AI as ambient infrastructure rather than a destination product.
- **n8n:** n8n is emerging as the orchestration backbone for AI‑driven workflows. By combining visual flows, code execution, and AI agents, it allows models, tools, and business processes to run together reliably. This makes it well‑suited for production automation where agents must interact with systems, not just users.
- **Supabase:** Supabase increasingly acts as the default backend for agentic apps, supplying auth, storage, databases, and edge functions. In AI systems, it often becomes the persistent memory and control plane that agents rely on to store state, manage users, and operate safely in production.
- **Firecrawl:** Firecrawl turns the web into LLM‑ready context by extracting clean, structured content instead of raw HTML. Its importance lies in enabling agents and RAG pipelines to work with live, external knowledge sources in a deterministic way, including via MCP integration.
- **awesome‑mcp‑servers:** This repository plays a crucial ecosystem role by cataloging MCP servers across domains like browsers, filesystems, data, and tools. It accelerates adoption by making MCP concrete and discoverable, helping shift agent development from bespoke integrations to standardized, governable interfaces.

**Title: OpenAI's AI data agent, built by two engineers, now serves thousands of employees — and the company says anyone can replicate it**

[Link](https://venturebeat.com/technology/openais-ai-data-agent-built-by-two-engineers-now-serves-4-000-employees-and)

Summary: Key take aways for my production agents:
- Data discovery is the real bottleneck: The hardest problem was not analysis but identifying the correct datasets among tens of thousands. Improving metadata, ownership clarity, and semantic indexing delivers more value than upgrading models.
- Force a deliberate discovery phase: The agent initially failed by acting too confidently and too fast. Explicitly prompting it to explore alternatives and validate sources before execution dramatically improved outcomes.
- Curated context beats maximal context: Providing less but higher‑quality context consistently outperformed dumping large volumes of raw schemas and documents. Precision matters more than recall in production agents.
- Horizontal scope unlocks real value: The agent succeeded by working across finance, product, and engineering data rather than being siloed. Cross‑domain answers enabled insights that isolated tools cannot produce.
- Layered context scales better than single‑shot RAG: Multiple context layers (schemas, expert notes, inferred relationships, institutional knowledge, live queries) made the agent robust and adaptable.
- Feedback persistence turns tools into memory: User corrections were stored and reused, allowing the agent to learn organizational knowledge over time instead of repeating mistakes.
- Transparency builds trust faster than accuracy alone: Streaming reasoning steps, exposing source choices, and enabling mid‑task interruption made users trust the agent even when it was imperfect.
- Simple guardrails outperform complex safety: Strong identity‑based access control and restricted write paths were sufficient, proving enterprise safety starts with permissions, not alignment tricks.

**Title: You don’t know what your agent will do until it’s in production**

[Link](https://www.langchain.com/blog/production-monitoring)

Summary: In production, monitor full prompt–response pairs, multi‑turn context, agent trajectories, tool calls, cost, latency, and quality signals - not just errors. Scale evaluation by combining targeted human review via annotation queues with sampled, automated LLM‑as‑judge evaluators. Production traces then feed a continuous loop: surface failure patterns, label and score them, turn them into evaluation datasets, test fixes, and validate improvements live.
