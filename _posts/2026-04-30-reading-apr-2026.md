**Title: Microsoft Copilot Studio Knowledge Source Limitations and Solutions**

[Link](https://github.com/Rickcau/Copilot-Studio/blob/main/Knowledge_Source_Limitations_Solutions/copilot-studio-knowledge-limitations-guide.md)

Summary: This guide explains why Copilot Studio's default knowledge sources are insufficient for production‑grade RAG. File uploads and the built‑in Azure AI Search connector hide critical retrieval controls (chunking, filtering, ranking), leading to poor answer quality and hard limits (payload size, no metadata filters). The document details what happens behind the scenes, why these constraints are structural (not bugs), and presents Custom Knowledge Sources or fully custom ingestion pipelines as the practical ways to regain control and reliability.

Key learnings:
- No chunking control = low RAG quality: Makers cannot set chunk size, overlap, or strategy; defaults are undocumented, often breaking semantic coherence.
- Minimal query configuration: You can’t choose keyword/vector/hybrid search, enable reranking, set $top, thresholds, field selection, or pagination in the built‑in connector.
- Payload size limits are permanent: Large Azure AI Search responses can exceed Copilot limits and fail; Microsoft confirms no fix is planned.
- No metadata filtering or security trimming: Even filterable fields in your index can’t be used, blocking versioning, scoping, or access control scenarios.
- Vectorization is mandatory but opaque: Auto "import & vectorize" wizards enforce defaults (models, text split) without maker control.
- Know the ingestion paths: File uploads go through Dataverse with hidden defaults; the Search connector is a thin pass‑through with no hooks.
- Use Custom Knowledge Sources for control: Custom endpoints let you own chunking, filtering, ranking, and payload shaping.
- Go fully custom when needed: For complex RAG, security trimming, or high scale, build your own ingestion/query pipeline outside Copilot Studio and integrate it.

**Title: Using Cheaper Reasoning Models Is Probably Costing You More!**

[Link](https://www.intoai.pub/p/price-reversal-phenomenon?utm_campaign=email-half-post&r=54qi6w&utm_source=substack&utm_medium=email)

Summary: 
The blog explains the **Price Reversal Phenomenon**, showing why cheaper‑listed reasoning models can end up costing more in real usage. While developers often choose LLMs based on advertised token prices, **actual inference cost is dominated by hidden "thinking" or reasoning tokens**. A large‑scale evaluation of frontier reasoning models across math, science, coding, and QA tasks reveals that in ~22% of comparisons, the model with the lower list price is more expensive overall—sometimes by orders of magnitude. The root cause is extreme variability in how much internal reasoning different models perform for the same query, with some consuming up to 9x more thinking tokens. Because this usage is noisy and hard to predict per request, API prices become an unreliable proxy for real cost. The article concludes that teams should select models using outcome‑level cost monitoring, not list prices, and should instrument production systems to track per‑query spend and efficiency, and evaluate cost from the evaluation tests.
